# 05. DATABASE — SI-PINTAR

## 1. Ringkasan

Dokumen ini menjelaskan rancangan database untuk **SI-PINTAR**.

Database digunakan untuk menyimpan data:

- User login.
- Role dan permission.
- Data karyawan.
- Departemen.
- Jabatan.
- Training.
- Materi training.
- Peserta training.
- Pre-test dan post-test.
- Soal dan opsi jawaban.
- Jawaban peserta.
- Attempt test.
- Penilaian essay.
- Progress training.
- Laporan training.

Database menggunakan konsep relasional dengan **MySQL** dan diimplementasikan pada **Laravel 13**.

------

## 2. Prinsip Database

Prinsip database SI-PINTAR:

1. Data login menggunakan tabel `users`.
2. Role hanya terdiri dari:
   - Admin
   - Karyawan
3. Permission menggunakan struktur role dan permission.
4. Data karyawan dipisah ke tabel `employees`.
5. Departemen dan jabatan menjadi master data.
6. Training memiliki materi, peserta, pre-test, dan post-test.
7. Pre-test dan post-test disimpan dalam satu tabel `tests`.
8. Soal pilihan ganda dan essay disimpan dalam tabel `questions`.
9. Opsi pilihan ganda disimpan dalam tabel `question_options`.
10. Jawaban peserta disimpan per attempt.
11. Essay dinilai manual di tabel jawaban.
12. Laporan tidak menggunakan tabel khusus.
13. Laporan diambil dari query data training, peserta, attempt, jawaban, dan nilai.
14. Default laporan menampilkan data bulan berjalan agar tidak membebani server.
15. Data penting menggunakan soft delete.
16. Hapus permanen tetap tersedia, tetapi wajib menampilkan modal konfirmasi/peringatan.

# 4. Relasi Antar Tabel

## 4.1 Relasi User dan Karyawan

```text
users
  └── employees
```

Keterangan:

- Satu user dengan role karyawan memiliki satu data employee.
- User dengan role admin tidak wajib memiliki data employee.

Relasi:

```text
users.id = employees.user_id
```

------

## 4.2 Relasi Master Data

```text
departments
  └── employees

positions
  └── employees
```

Relasi:

```text
departments.id = employees.department_id
positions.id = employees.position_id
```

Keterangan:

- Satu departemen dapat memiliki banyak karyawan.
- Satu jabatan dapat memiliki banyak karyawan.

------

## 4.3 Relasi Training

```text
trainings
  ├── training_materials
  ├── training_participants
  └── tests
```

Relasi:

```text
trainings.id = training_materials.training_id
trainings.id = training_participants.training_id
trainings.id = tests.training_id
```

Keterangan:

- Satu training dapat memiliki banyak materi.
- Satu training dapat memiliki banyak peserta.
- Satu training dapat memiliki pre-test dan post-test.

------

## 4.4 Relasi Peserta Training

```text
employees
  └── training_participants
        └── trainings
```

Relasi:

```text
employees.id = training_participants.employee_id
trainings.id = training_participants.training_id
```

Keterangan:

- Satu karyawan dapat mengikuti banyak training.
- Satu training dapat memiliki banyak karyawan.
- Kombinasi `training_id` dan `employee_id` harus unique.

------

## 4.5 Relasi Materi dan Akses Materi

```text
training_materials
  └── material_access_logs
        └── employees
```

Relasi:

```text
training_materials.id = material_access_logs.training_material_id
employees.id = material_access_logs.employee_id
```

Keterangan:

- Akses materi dicatat per materi.
- Riwayat akses materi otomatis tersimpan.
- Post-test terbuka setelah materi wajib diakses.

------

## 4.6 Relasi Test, Soal, dan Opsi

```text
tests
  └── questions
        └── question_options
```

Relasi:

```text
tests.id = questions.test_id
questions.id = question_options.question_id
```

Keterangan:

- Pre-test dan post-test disimpan dalam tabel `tests`.
- Perbedaan jenis test ditentukan dari field `type`.
- Soal bisa berupa pilihan ganda atau essay.
- Opsi jawaban hanya digunakan untuk soal pilihan ganda.

------

## 4.7 Relasi Attempt dan Jawaban

```text
test_attempts
  └── test_answers
        ├── questions
        └── question_options
```

Relasi:

```text
test_attempts.id = test_answers.test_attempt_id
questions.id = test_answers.question_id
question_options.id = test_answers.selected_option_id
```

Keterangan:

- Jawaban peserta disimpan per attempt.
- Jawaban tidak dioverwrite.
- Jika peserta mengulang post-test, sistem membuat attempt baru.
- Essay dinilai manual pada tabel `test_answers`.

# 8. Soft Delete dan Delete Permanen

## 8.1 Tabel yang Menggunakan Soft Delete

Tabel yang menggunakan soft delete:

```text
users
employees
departments
positions
trainings
training_materials
tests
questions
```

Tujuan soft delete:

- Menjaga histori data.
- Menghindari laporan rusak.
- Menghindari kehilangan data penting.
- Memungkinkan data dipulihkan jika dibutuhkan.

------

## 8.2 Tabel yang Tidak Wajib Soft Delete

Tabel yang tidak wajib soft delete:

```text
training_participants
material_access_logs
question_options
test_attempts
test_answers
roles
permissions
```

Catatan:

- `test_attempts` dan `test_answers` merupakan data histori.
- Data histori tidak disarankan dihapus kecuali admin benar-benar melakukan hapus permanen dengan konfirmasi.

------

## 8.3 Rule Hapus Data

Rule hapus data:

1. Semua aksi hapus wajib menggunakan custom modal.
2. Tidak boleh menggunakan `alert()` atau `confirm()` bawaan browser.
3. Modal harus menjelaskan dampak penghapusan.
4. Hapus permanen tetap boleh dilakukan.
5. Sistem wajib menampilkan peringatan jika data memiliki relasi.
6. Admin harus klik tombol konfirmasi sebelum data dihapus.

Contoh pesan modal:

```text
Hapus Data Permanen?

Data ini mungkin terhubung dengan data training, peserta, nilai, atau laporan.
Jika dihapus permanen, data tidak dapat dikembalikan.

[ Batal ] [ Hapus Permanen ]
```

# 15. Catatan Implementasi Laravel

Rekomendasi implementasi Laravel:

1. Gunakan migration untuk semua tabel.
2. Gunakan foreign key constraint.
3. Gunakan enum string agar mudah dibaca.
4. Gunakan soft delete untuk data utama.
5. Gunakan seeder untuk role, permission, admin, dan dummy data.
6. Gunakan model relationship yang jelas.
7. Gunakan policy atau middleware untuk akses role.
8. Gunakan pagination pada semua table.
9. Gunakan filter default bulan berjalan pada laporan.
10. Gunakan query builder atau Eloquent relation untuk laporan.
11. Jangan menyimpan laporan sebagai tabel khusus pada MVP.
12. Jangan menampilkan kunci jawaban kepada karyawan.
