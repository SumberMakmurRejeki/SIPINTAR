# 03. UI/UX Specification — SI-PINTAR

## 1. Ringkasan

SI-PINTAR menggunakan konsep desain **Modern SaaS Dashboard** dan **Minimalist Enterprise System**.

Desain sistem harus:

- Modern.
- Minimalist.
- Clean.
- Responsive.
- Mudah digunakan admin dan karyawan.
- Konsisten antar halaman.
- Menggunakan komponen UI reusable.
- Menggunakan icon, bukan emoji.
- Tidak menggunakan konfirmasi bawaan browser seperti `alert()` dan `confirm()`.
- Dapat diakses dengan nyaman di desktop, tablet, dan mobile.

------

## 2. Konsep Desain Sistem

Konsep utama desain SI-PINTAR adalah:

```text
Modern SaaS Dashboard
Minimalist Enterprise System
Tailwind UI Inspired
Clean Internal Training Platform
```

Karakter desain:

- Tampilan dashboard modern.
- Layout bersih dan tidak ramai.
- Fokus pada data, status, dan aksi utama.
- Warna tidak berlebihan.
- Komponen konsisten dan reusable.
- Setiap halaman memiliki struktur yang mudah dipahami.
- Karyawan diarahkan dengan satu tombol utama sesuai status training.

------

## 3. Gaya Visual

## 3.1 Style Utama

Gaya visual sistem:

- Modern.
- Minimalist.
- Corporate clean.
- SaaS dashboard.
- Responsive.
- Professional.
- Tidak dekoratif berlebihan.

## 3.2 Prinsip Visual

- Gunakan whitespace yang cukup.
- Gunakan border tipis untuk pemisah section.
- Gunakan shadow ringan hanya untuk card/modal penting.
- Gunakan icon untuk membantu pemahaman.
- Jangan menggunakan emoji.
- Jangan terlalu banyak warna dalam satu halaman.
- Gunakan status badge agar data mudah dibaca.

## 4.1 Color Palette

| Token          | Warna      | Kode      | Penggunaan                          |
| -------------- | ---------- | --------- | ----------------------------------- |
| Primary        | Near Black | `#080808` | Tombol utama, teks penting, heading |
| Background     | White      | `#FFFFFF` | Background utama                    |
| Surface        | White      | `#FFFFFF` | Card, modal, form                   |
| Border         | Light Gray | `#D8D8D8` | Border input, card, table           |
| Text Primary   | Black      | `#080808` | Judul dan teks utama                |
| Text Secondary | Gray       | `#5A5A5A` | Deskripsi dan helper text           |
| Muted          | Soft Gray  | `#898989` | Placeholder, teks tambahan          |
| Info           | Blue       | `#146EF5` | Info, link, status berjalan         |
| Success        | Green      | `#00A65A` | Status berhasil/lulus               |
| Warning        | Yellow     | `#FFAE13` | Menunggu penilaian/peringatan       |
| Danger         | Red        | `#EE1D36` | Error, hapus, tidak lulus           |
| Orange         | Orange     | `#FF6B00` | Status mengulang                    |

## 4.2 Aturan Penggunaan Warna

- Primary button menggunakan warna `#080808`.
- Warna aksen tidak digunakan sembarangan untuk tombol utama.
- Warna aksen digunakan untuk:
  - badge status,
  - alert,
  - chart,
  - highlight card,
  - indikator progress.
- Warna merah hanya untuk aksi berbahaya atau error.
- Warna hijau hanya untuk status sukses/lulus.
- Warna kuning hanya untuk warning atau menunggu penilaian.

## 5. Tipografi

## 5.1 Font

Font utama sistem:

```text
Inter, system-ui, sans-serif
```

Alasan:

- Modern.
- Clean.
- Mudah dibaca.
- Cocok untuk dashboard.
- Cocok dengan Tailwind CSS.

## 5.2 Skala Font

| Elemen        | Ukuran      | Weight  | Penggunaan                   |
| ------------- | ----------- | ------- | ---------------------------- |
| Page Title    | 24px        | 600     | Judul halaman                |
| Section Title | 18px        | 600     | Judul section/card           |
| Body Text     | 14px - 16px | 400     | Teks utama                   |
| Table Text    | 14px        | 400     | Isi table                    |
| Button Text   | 14px - 16px | 500     | Label tombol                 |
| Caption       | 12px        | 400/500 | Helper text, badge, metadata |

## 5.3 Aturan Tipografi

- Heading tidak boleh terlalu besar.
- Gunakan maksimal font weight 600 untuk menjaga tampilan clean.
- Body text harus mudah dibaca.
- Helper text harus lebih kecil dan berwarna muted.
- Label form menggunakan ukuran 14px dan weight 500.


## 7.2 Table

Table digunakan untuk halaman admin dan laporan.

Table desktop/tablet:

- Header jelas.
- Row memiliki border tipis.
- Action berada di kolom kanan.
- Search dan filter berada di atas table.
- Pagination berada di bawah table.

Table mobile:

- Table berubah menjadi card list.
- Setiap row menjadi satu card.
- Kolom penting ditampilkan sebagai label-value.

## 7.3 Form

Form digunakan untuk create dan edit data.

Aturan form:

- Form dibagi menjadi beberapa section.
- Jangan membuat form terlalu panjang tanpa pembagian.
- Gunakan label yang jelas.
- Gunakan helper text jika perlu.
- Error validasi tampil di bawah input.
- Tombol simpan berada di area bawah form.

## 7.4 Modal

Modal digunakan untuk:

- Konfirmasi hapus.
- Konfirmasi publish.
- Konfirmasi tutup training.
- Form kecil jika diperlukan.
- Preview singkat jika diperlukan.

Aturan modal:

- Wajib custom modal.
- Tidak boleh menggunakan `alert()`.
- Tidak boleh menggunakan `confirm()` bawaan browser.
- Harus memiliki tombol batal.
- Aksi berbahaya menggunakan warna danger.

## 10. State Tampilan

## 10.1 Empty State

Empty state digunakan saat data belum tersedia.

Struktur empty state:

```text
Icon
Judul
Deskripsi
Tombol aksi jika diperlukan
```

Contoh:

```text
Belum ada training
Tambahkan training pertama untuk mulai mengelola pelatihan karyawan.
[Tambah Training]
```

## 10.2 Loading State

Loading state digunakan saat sistem mengambil atau menyimpan data.

Jenis loading:

- Spinner kecil pada button.
- Skeleton row pada table.
- Skeleton block pada card.
- Loading overlay ringan untuk proses tertentu.

Contoh button loading:

```text
Menyimpan...
Memproses...
Mengexport...
```

## 10.3 Error State

Error state harus jelas dan mudah dipahami.

Contoh:

```text
Gagal menyimpan data. Periksa kembali field yang wajib diisi.
File materi terlalu besar. Maksimal ukuran file adalah 20 MB.
Password baru dan konfirmasi password tidak sama.
```

## 10.4 Success State

Success state digunakan setelah aksi berhasil.

Contoh:

```text
Data berhasil disimpan.
Training berhasil dipublish.
Nilai essay berhasil disimpan.
Laporan berhasil diexport.
```

Success state biasanya ditampilkan melalui toast.

## 10.5 Validation Error

Validation error tampil inline di bawah field.

Contoh:

```text
Nama karyawan wajib diisi.
Tanggal selesai tidak boleh lebih awal dari tanggal mulai.
Passing grade harus berupa angka 0 sampai 100.
```

Aturan:

- Input error menggunakan border merah.
- Pesan error jelas.
- Jangan hanya menampilkan error umum di atas form.
- Field yang salah harus mudah ditemukan.

## 11. Responsive Behavior

## 11.1 Desktop

Breakpoint:

```text
>= 1024px
```

Behavior:

- Sidebar kiri tampil.
- Sidebar bisa collapse ke icon-only.
- Navbar atas tampil penuh.
- Table tampil normal.
- Dashboard menggunakan grid 3-4 kolom.
- Form bisa 2 kolom jika field tidak terlalu panjang.
- Report table menggunakan full width.

## 11.2 Tablet

Breakpoint:

```text
768px - 1023px
```

Behavior:

- Sidebar bisa collapse.
- Grid dashboard menjadi 2 kolom.
- Form maksimal 2 kolom.
- Table masih bisa digunakan dengan horizontal scroll ringan.
- Tabs bisa horizontal scroll jika terlalu panjang.

## 11.3 Mobile

Breakpoint:

```text
< 768px
```

Behavior:

- Sidebar menjadi off-canvas drawer.
- Toggle sidebar berada di navbar.
- Content area full width.
- Dashboard card menjadi 1 kolom.
- Table berubah menjadi card list.
- Form menjadi 1 kolom.
- Tombol action full width jika diperlukan.
- Modal menyesuaikan lebar layar.
- Tabs menjadi horizontal scroll.
- Nomor soal test menjadi horizontal scroll atau drawer.

## 12. UX Rules

## 12.1 Konfirmasi Sebelum Hapus

Semua aksi hapus wajib menggunakan custom modal.

Tidak boleh menggunakan:

```text
alert()
confirm()
prompt()
```

Contoh modal hapus:

```text
Judul:
Hapus Data?

Deskripsi:
Data yang dihapus tidak dapat dikembalikan.

Tombol:
Batal
Hapus Permanen
```

## 12.2 Pesan Sukses Setelah Simpan

Setiap aksi berhasil harus memberi feedback.

Contoh:

```text
Data berhasil disimpan.
Data berhasil diperbarui.
Data berhasil dihapus.
Nilai berhasil disimpan.
```

## 12.3 Error Harus Jelas

Error harus menjelaskan masalah dan tindakan yang perlu dilakukan.

Contoh buruk:

```text
Error.
```

Contoh baik:

```text
Gagal menyimpan data. Periksa kembali field yang wajib diisi.
```

## 12.4 Form Tidak Terlalu Panjang

Form panjang harus dibagi menjadi section.

Contoh section:

```text
Informasi Utama
Pengaturan Test
Status & Akses
Catatan Opsional
```

## 12.5 Tombol Aksi Harus Konsisten

Aturan posisi tombol:

```text
Primary action: kanan
Secondary action: kiri atau sebelah primary
Danger action: warna merah dan wajib modal
```

Contoh:

```text
[Batal] [Simpan]
[Batal] [Hapus Permanen]
```

## 12.6 Karyawan Mengikuti Satu Tombol Utama

Karyawan tidak perlu memilih alur secara manual.

Tombol utama berdasarkan status:

| Kondisi                              | Tombol             |
| ------------------------------------ | ------------------ |
| Belum mengerjakan pre-test           | Kerjakan Pre-Test  |
| Pre-test selesai                     | Buka Materi        |
| Materi sudah diakses                 | Kerjakan Post-Test |
| Ada essay belum dinilai              | Menunggu Penilaian |
| Training selesai                     | Lihat Hasil        |
| Tidak lulus dan masih bisa mengulang | Ulangi Test        |

## 12.7 Aksi Berbahaya Harus Dipisahkan

Aksi seperti hapus permanen tidak boleh terlalu dekat dengan tombol utama.

Aturan:

- Gunakan warna danger.
- Gunakan custom confirmation modal.
- Tulis dampak aksi dengan jelas.
- Jangan menggunakan icon saja tanpa label pada aksi penting.

Aturan accessibility:

- Kontras warna harus jelas.
- Tombol minimal mudah ditekan di mobile.
- Input harus memiliki label.
- Icon penting harus memiliki label atau tooltip.
- Error tidak hanya bergantung pada warna.
- Fokus keyboard harus terlihat.
- Modal dapat ditutup dengan tombol batal atau close.
- Tidak menggunakan teks terlalu kecil untuk informasi penting.
